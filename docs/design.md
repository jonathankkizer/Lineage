# A Mac-Assed dbt Graph Viewer

## The pitch

A native macOS app for exploring a dbt project as a graph. The web representations — dbt docs' built-in viewer, dbt Cloud's lineage view, Dagster's asset graph — are functional but distinctly web-flavored: laggy pan/zoom, weak selection models, no real keyboard navigation, no platform integration. A native Mac app can be dramatically better at the things that matter: 120Hz trackpad-driven pan/zoom, real selection and focus models, keyboard-first navigation, Quick Look previews, Mac-native window management. The graph is the right showcase for native rendering — sparse enough to lay out cleanly, dense enough to be interesting, semantically rich enough that hover and selection states have meaningful content.

The target user is the dbt practitioner who prefers Mac-assed Mac apps (BBEdit, Nova, Transmit) over Electron and the browser, and who finds dbt docs' graph view unusable past ~100 nodes. Niche, but a real niche, and the dogfooding loop is tight: this gets used daily.

## Why this gap exists

dbt's graph visualization story is owned by web frontends:

- **dbt docs (`dbt docs serve`)** — d3 force-directed layout, becomes unusable past a few hundred nodes. No keyboard navigation. Pan/zoom feels web-canvas-y.
- **dbt Cloud lineage view** — better than dbt docs but still browser-bound. Subscription gated.
- **Dagster asset graph** — the best of the bunch but still a browser tab, and only relevant if you're on Dagster.
- **VS Code "Power User for dbt" extension and similar** — useful for VS Code users; nothing for people who prefer real Mac editors.

Nobody has built a native Mac graph viewer because the audience overlap (dbt users who care about Mac native craft) is small. That's fine — this is a personal project that happens to be defensible as a small indie app.

## Why this works as a first AppKit project

A few project-shape advantages worth being explicit about:

- **No auth, no networking, no credential management.** Everything is local file parsing.
- **No multi-provider abstraction.** One data format (dbt artifacts), one mental model.
- **The hard part is the fun part.** Graph rendering and interaction design is where platform craft and interesting engineering coincide. Not 800 lines of boilerplate to get to a payoff.
- **The architecture fits in one head.** Parse JSON → build graph → render and interact.
- **Audience of one for as long as needed.** Build it for yourself; expose it more broadly only if it earns that.
- **Failure is cheap.** A few weekends, not a few months, to know whether AppKit dev is fun again.

The AppKit muscles built here — NSDocument, NSWindow, NSToolbar, NSViewController, custom CALayer-based view rendering, inspector panes, preferences, menus, keyboard navigation, drag-and-drop, Quick Look, FSEvents — transfer directly to any future Mac project.

## The data model: dbt artifacts

When dbt runs (compile, run, build, parse, etc.), it writes JSON files into `target/` in the project root. All local, well-documented schemas, no network calls.

**Primary inputs:**

- **`manifest.json`** — the graph. Every model, source, seed, snapshot, test, exposure, metric, macro, with `depends_on.nodes` arrays giving edges. Includes file paths, configs, descriptions, columns, tags, materializations, schemas, database/schema targets.
- **`run_results.json`** — most recent run outcomes: status, runtime, error messages per node. Per-invocation, overwritten on each run.
- **`catalog.json`** — generated by `dbt docs generate`. Warehouse-side metadata: actual column types, row counts where available, table sizes. Joins to manifest by node ID.
- **`sources.json`** — freshness check results if `dbt source freshness` has been run.
- **`semantic_manifest.json`** — MetricFlow / semantic layer definitions if the project uses them.

**Secondary inputs:**

- **`target/compiled/<project>/...`** — compiled SQL per node (Jinja resolved, no warehouse-specific transformations).
- **`target/run/<project>/...`** — final SQL as sent to the warehouse.

The dependency graph is built directly from `manifest.json` — every node has `depends_on.nodes` listing its parents. Build a directed graph from that and the data layer is done.

## What "Mac-assed" means for this app

Non-negotiable platform behaviors. The whole reason this app exists rather than another web view.

### Framework choices

- **AppKit only.** No SwiftUI. The apps universally cited as Mac-assed (Transmit, BBEdit, Acorn, OmniFocus, Tower, Xcode itself) are AppKit. SwiftUI on the Mac is still incomplete and retrofitted from iOS; AppKit was designed for the Mac from the start.
- **Modern Swift wrapped around AppKit.** Swift 6, strict concurrency, async/await throughout.
- **Programmatic UI, no XIBs or Storyboards.** Auto Layout in code, NSStackView for composition. Easier to diff, refactor, and edit with LLM assistance.
- **Target macOS 26+.** Latest APIs everywhere. No back-compat overhead for a hobby project.

### Document architecture

- **NSDocument.** A dbt project is the document. Each open project is an NSDocument instance with its own window. Opening a project means picking the project root (the directory containing `dbt_project.yml`); the app locates `target/` from there.
- **Recent documents** in the File menu, working correctly with the standard Apple Events plumbing.
- **Multiple windows for multiple projects** — open Suvida's analytics repo and an experiment repo side by side.
- **Tabs within a window** for multiple views of the same project (e.g., full graph in one tab, focused lineage on a specific model in another).
- **Window state restoration** on relaunch. Open documents, window positions, zoom level, selection, filter state all restore.

### Platform integration

- Proper menu bar with full standard menus: File, Edit, View, Navigate, Window, Help. Menu item validation through the responder chain. Keyboard shortcuts on every reasonable action.
- Real NSToolbar with customizable items (layout mode picker, filter field, run status toggle, etc.).
- Drag-and-drop. Drag a project folder onto the dock icon or app window to open. Drag a node out as a URL (or as a file reference to its `.sql`).
- Quick Look integration. Space bar on a selected node previews the compiled SQL, description, columns, recent run times — Finder-style.
- Services menu integration.
- AppleScript dictionary. Real scripting support.
- Shortcuts app integration via App Intents.
- Dark mode that works correctly across all accent colors.
- VoiceOver and full keyboard accessibility.

### Preferences

- Classic Mac preference window: NSToolbar across the top with icon+label items per pane, smooth animated resize between panes, title updates to selected pane, non-resizable, centered, remembers last selected pane, Cmd+, opens it.
- **Explicitly not the SwiftUI Settings scene API.** System Settings.app in Ventura+ is the cautionary tale.

## Architecture

### Project structure

```
DbtViewer/
├── App/                      # AppDelegate, app-level menu, document controller
├── Document/                 # NSDocument subclass, project loading
├── Model/                    # Parsed manifest, graph types, run results
├── Graph/                    # Layout algorithms, graph traversal utilities
├── Rendering/                # CALayer-based graph view, hit testing, animation
├── UI/                       # Window controller, toolbar, inspector, filter bar
├── QuickLook/                # Quick Look preview generation
└── Preferences/              # Preference panes
```

### Data layer

- **Parse on background queue.** `manifest.json` for a real project is several MB and parsing on the main thread will jank window opening. Use `JSONDecoder` with `Codable` types, run in a `Task.detached`.
- **In-memory graph representation.** Adjacency lists for forward and reverse edges (you'll need both — "what does this depend on" and "what depends on this" are both critical queries). Node IDs as the canonical key.
- **Lazy enrichment.** Load `manifest.json` first to draw the graph fast; merge `catalog.json` and `run_results.json` in afterward as they become relevant.
- **FSEvents watcher on `target/`.** When files change (dbt was re-run in another terminal), reload and re-render. Debounce to avoid thrashing during a `dbt build`.

### Layout

- **Sugiyama-style layered layout** as the default. Layer assignment by longest-path from roots, crossing minimization within layers, x-coordinate assignment. Right tradeoff of "looks like a DAG" and "doesn't take forever to compute" for graphs in the 100-2000 node range.
- **Layout off the main thread.** Compute on a background queue, deliver positions back to the main thread to apply. Cache the result keyed by manifest hash so re-opening a project is instant.
- **Multiple layouts in v2:** layered (default), focus-on-node (N hops up/down from selection), lineage (linearized ancestors → node → descendants), grouped-by-schema, grouped-by-tag.
- **No third-party layout dependency for v1.** A hand-written Sugiyama in Swift is a meaningful but bounded chunk of work, and avoids a Graphviz/libcgraph dependency. Worth doing.

### Rendering

- **CALayer per node, CAShapeLayer per edge.** Sweet spot for the scale that matters (up to ~2000 nodes). Hit testing is straightforward, animation is free (implicit animations on layer property changes), and you get GPU compositing without writing Metal.
- **Don't render in NSView's `draw(_:)`.** That path works up to ~200 nodes and then degrades. CALayer composition is the right tool.
- **Metal as a v3+ consideration only.** Overkill for the actual use case. Only revisit if real-world projects with 5000+ nodes become a target.
- **Smooth pan/zoom via gesture recognizers on a containing NSView.** Magnification gesture for zoom, scroll wheel / two-finger pan for pan. The trackpad inertia and rubberbanding macOS users expect comes free from doing this correctly with `NSScrollView` or a hand-rolled equivalent.
- **Level-of-detail rendering.** At zoomed-out scales, draw simplified node representations (just a colored rect, no text). As the user zooms in, layers gain labels, then icons, then column counts. Keeps frame rates high at any zoom level.

### Interaction model

- **Selection.** Click to select, Shift+click to extend, Cmd+click to toggle, marquee drag to select a region. Selection drives the inspector pane.
- **Keyboard navigation.** Arrow keys traverse edges from the focused node (up = parents, down = children, left/right = siblings at same layer). Focused node always animates to stay in view. Tab cycles through search results. Cmd+F focuses the filter field. Escape clears selection.
- **Hover states.** Hovering a node highlights its immediate parents and children with a different color; clicking pins the highlight. Standard graph viewer affordance, done with native animation timing.
- **Double-click a node to "focus"** — re-layout the graph centered on that node showing N hops up/down (configurable). Cmd+] / Cmd+[ to traverse focus history (like browser forward/back).
- **Right-click context menu** with node-relevant actions: Reveal in Finder, Copy Node ID, Copy Compiled SQL, Show Lineage.

### Inspector pane

- Cmd+Opt+I toggles. Right-side pane, resizable.
- Shows full node detail for the selected node: type (model/source/test/etc.), description, file path (clickable to reveal in Finder), materialization, schema, database, tags, columns (with types from catalog if available), depends-on list, referenced-by list, recent run status and runtime if `run_results.json` present.
- "Open in editor" button — uses the user's configured editor (BBEdit, Nova, Cursor, VS Code, etc.) via `NSWorkspace.open` or `x-callback-url` schemes where supported.

### Filter and search

- **Filter bar** above the graph. Cmd+F focuses it.
- **Query language is simple text by default.** Type "stg_" and the graph dims everything not matching `stg_*`. As complexity is needed, support qualifiers: `tag:finance`, `materialization:incremental`, `schema:analytics`, `status:failed`, `modified:24h`.
- **Filter dims rather than hides.** Matching nodes stay vivid, non-matching nodes dim to ~20% opacity. The path between matches stays visible — critical for understanding lineage.

## Scoping

### v1 — useful to you, sufficient to ship

The bar: replaces dbt docs' graph view for your daily Suvida work.

- Open a dbt project (pick the project root; app finds `target/manifest.json`)
- Parse `manifest.json` and render the DAG with layered layout
- Smooth trackpad pan/zoom
- Click to select; inspector pane with full node detail
- Keyboard navigation: arrow keys traverse edges, focused node stays in view
- Filter bar with simple text matching plus a few qualifiers (tag:, materialization:, schema:)
- Color/badge nodes by run status if `run_results.json` is present
- Quick Look preview (Space bar) showing compiled SQL and node metadata
- FSEvents-based reload when `target/` changes
- Proper Mac shell: NSDocument architecture, menus, toolbar, window restoration, classic preference window
- Recent documents, multi-project support via multiple windows

### v2 — meaningful upgrades

- Focus mode: re-layout centered on a node, N hops up/down, with history (Cmd+] / Cmd+[)
- Lineage mode: linearized ancestors → node → descendants
- Multiple layout algorithms with a toolbar picker
- Catalog integration: column types, row counts, table sizes from `catalog.json`
- Compiled SQL viewer pane (read from `target/compiled/`)
- Tabs within a window for multiple views of the same project
- "Open in editor" with configurable editor preference
- Diff between two manifest snapshots (useful for PR review: what changed in the DAG?)

### v3 — if still having fun

- Test result visualization (which tests are failing, where)
- Semantic layer / MetricFlow node types treated as first-class
- Exposures and metrics rendered distinctly
- Local history of run results (capture each `run_results.json` snapshot) with runtime trend overlays on nodes
- Cost overlays if the warehouse provides them via a sidecar file or annotation
- Export current view as SVG/PNG for documentation

### Explicit non-goals

- **Editing dbt files.** Not building an IDE. The user edits in their preferred editor; this app reflects the artifacts.
- **Running dbt commands.** No "Run" button in v1. The user runs dbt in their terminal; the app watches the artifacts. (A "Run dbt build" shell-out could be added later, but resist it for v1 — dramatically expands scope into process management, log streaming, etc.)
- **Warehouse connectivity.** No live queries. Everything from local files.
- **dbt Cloud API integration.** Local-only is the premise. Maybe revisit later.
- **Cross-platform.** Mac-only.

## Risks and known hard parts

- **AppKit + modern Swift skill ramp.** Coming back to Apple platform dev after ~10 years. LLM assistance is weaker on AppKit than on web frameworks; training data skews older (Objective-C, IBOutlets, completion handlers). Mitigation: read NetNewsWire source (open-source, modern AppKit Swift) before starting; be explicit in prompts about Swift 6, async/await, programmatic UI, no XIBs; modernize LLM output rather than trusting it.
- **Sugiyama layout from scratch.** Bounded but non-trivial. Plan for a weekend or two just on layout, with a known-good test fixture (a small hand-built DAG) to validate against.
- **CALayer performance tuning.** Naive layer-per-node works to a point. Beyond that, group small layers, simplify shadows, drop labels at low zoom levels. Profile with Instruments before optimizing.
- **Notarization, sandboxing, code signing.** Less glamorous, time-consuming, LLM assistance is weakest here. Budget real time.
- **The Mac-assed bar is high and unforgiving.** A toolbar with the wrong style, a pan gesture that doesn't match Finder's inertia, a preference pane that snaps instead of animating — Mac users notice in seconds. The trained eye comes from studying Apple's own apps and Apple's own sample code, not from generated output.

## North star

The graph view in dbt docs but native. Smooth at any project size that matters. Keyboard-first. The kind of app where a power user can move through a 500-node DAG faster than they can in any web tool, and where the interaction feels like Finder or Nova rather than a browser tab.

Ship v1 when it replaces `dbt docs serve` for daily Suvida work. Everything after that is upside — and a much smaller, more honest commitment than the blob storage app while still building all the AppKit muscles that project would need.
